# FastAPI 封装统一成功响应格式

## 一、为什么需要统一响应格式

没有统一封装之前，每个接口返回的格式五花八门：

```python
# 接口A 返回字典
return {"code": 200, "data": user_info}

# 接口B 返回 Pydantic 模型
return UserInfoResponse(id=1, username="foo")

# 接口C 返回 ORM 对象
return db_user
```

前端对接困难，不知道每个接口的结构是否一致。

**目标**：所有接口统一返回 `{ "code": 200, "message": "...", "data": ... }`。

---

## 二、定义统一外壳函数

```python
# utils/response.py

from fastapi.responses import JSONResponse
from fastapi.encoders import jsonable_encoder


def success_response(message: str = "success", data=None):
    content = {
        "code": 200,
        "message": message,
        "data": data,
    }
    return JSONResponse(content=jsonable_encoder(content))
```

---

## 三、问题：data 的类型不统一

调用时 data 可能是各种东西：

```python
# 普通字典
success_response(data={"id": 1, "name": "foo"})

# Pydantic 模型
success_response(data=UserInfoResponse(id=1, username="foo"))

# SQLAlchemy ORM 对象
success_response(data=db_user)

# datetime / UUID / Decimal 等
success_response(data=created_at)
```

JSON 的 `json.dumps()` 不认识这些类型，直接返回会报错：

```
TypeError: Object of type datetime is not JSON serializable
```

---

## 四、jsonable_encoder 解决序列化问题

### 先搞清楚问题：什么是「序列化」？

**序列化** = 把 Python 对象变成 JSON 字符串的过程。

```text
Python 对象（内存中）             JSON 字符串（网络上传输）
  {"id": 1, "name": "Alice"}  →  '{"id": 1, "name": "Alice"}'
```

Python 内置的 `json.dumps()` **只能处理 7 种基本类型**：

```python
import json

# ✅ json.dumps 能处理的：str, int, float, bool, None, list, dict
json.dumps({"name": "Alice", "age": 25})   # ✅ 正常
json.dumps([1, 2, 3])                      # ✅ 正常

# ❌ json.dumps 不能处理的：
json.dumps({"time": datetime.now()})       # ❌ TypeError: datetime 不是 JSON 原生类型
json.dumps({"id": UUID("...")})            # ❌ TypeError: UUID 不是 JSON 原生类型
json.dumps({"price": Decimal("19.99")})    # ❌ TypeError: Decimal 不是 JSON 原生类型
json.dumps(user_pydantic_model)            # ❌ TypeError: Pydantic 模型不是 JSON 原生类型
```

而你项目里的数据**到处都是这些特殊类型**：

```python
class NewsItemResponse(BaseModel):
    publish_time: datetime    # ← datetime，JSON 不认识
    id: int
    title: str
```

### jsonable_encoder 做了什么？

`jsonable_encoder` 是 FastAPI 提供的一个**递归转换工具**，在调用 `json.dumps()` **之前**，先把所有特殊类型转成 JSON 认识的类型：

```python
from fastapi.encoders import jsonable_encoder

data = {
    "name": "Alice",
    "created_at": datetime(2026, 5, 25, 12, 0, 0),
    "price": Decimal("19.99"),
    "tags": {"a", "b"},            # set，JSON 也不认识
}

encoded = jsonable_encoder(data)
# encoded 现在变成：
# {
#     "name": "Alice",
#     "created_at": "2026-05-25T12:00:00",   ← datetime → 字符串
#     "price": 19.99,                         ← Decimal → float
#     "tags": ["a", "b"],                     ← set → list
# }

# 现在 json.dumps 就能处理了
json.dumps(encoded)  # ✅ 100% 不会报错
```

**转换规则（通俗版）：**

| Python 中的类型 | JSON 能认识吗 | jsonable_encoder 转成什么 |
|---------------|-------------|------------------------|
| `str`, `int`, `float`, `bool`, `None` | ✅ 认识 | 不变 |
| `list`, `dict` | ✅ 认识 | 不变，但递归处理里面的元素 |
| `datetime` | ❌ 不认识 | `"2026-05-25T12:00:00"`（ISO 格式字符串） |
| `UUID` | ❌ 不认识 | `"550e8400-e29b-..."`（字符串） |
| `Decimal` | ❌ 不认识 | `19.99`（float） |
| `set` | ❌ 不认识 | `[1, 2, 3]`（list） |
| Pydantic 模型 | ❌ 不认识 | `{"id": 1, "name": "Alice"}`（dict） |
| ORM 对象 | ❌ 不认识 | `{"id": 1, "username": "jack"}`（dict） |
| 嵌套对象 | ❌ 不认识 | 递归全部展开 |

### 为什么 JSONResponse 不会报错？

```python
# JSONResponse 内部做的是：
content = {"code": 200, "data": datetime.now()}
json.dumps(content)     # ❌ 直接报错！

# 所以你要先转：
encoded = jsonable_encoder(content)
# encoded = {"code": 200, "data": "2026-05-25T12:00:00"}

JSONResponse(content=encoded)   # ✅ 先转再传
# JSONResponse 内部：json.dumps(encoded) → 不会报错

# 简写成一行的完整流程：
JSONResponse(content=jsonable_encoder(content))
#                  ↑ 递归把所有特殊类型转成 JSON 认识的基本类型
#              ↑ 然后再序列化成 JSON 字符串
```

### FastAPI 正常返回时为什么不需要手动调用？

```python
# 当你这样返回时
@app.get("/user")
def get_user():
    return UserInfoResponse(id=1, username="Alice")
#            ↑ FastAPI 内部自动调了 jsonable_encoder
#            所以不需要你手动调
```

**但你的 `success_response` 里面包了一层 `{code, message, data}`**，是手动构造的字典，所以需要自己调 `jsonable_encoder`。

## 五、Pydantic model_validate：ORM 对象 → Pydantic 模型

### 先搞清楚问题：为什么需要 Pydantic 响应模型？

你从数据库拿到的数据是 ORM 对象（SQLAlchemy）：

```python
user = db.query(User).first()     # user 是 SQLAlchemy 的 User 对象
# user.id = 1
# user.username = "jack"
# user.password = "123456"         # ← 有密码！
# user.created_at = datetime(...)  # ← 数据库字段全都有
```

你不能直接把这个对象返回给前端：
1. **有敏感字段** — `password` 会被发出去
2. **有无关字段** — `created_at`、`updated_at` 前端不需要
3. **格式不统一** — 不同 ORM 返回的格式不一致

**Pydantic 模型**就是一道「过滤网」：只保留你声明了的字段，自动处理类型转换。

```python
class UserInfoResponse(BaseModel):
    id: int
    username: str
    # 故意没有 password、created_at 等字段
```

### 问题：ORM 对象不能直接塞给 Pydantic

```python
# ORM 对象
user = User(id=1, username="jack", nickname="杰克", password="123456")

# 你想转成 Pydantic 模型 → 过滤掉 password 等字段
UserInfoResponse(user)
# ❌ 报错：UserInfoResponse() 只接受关键字参数或字典
#    不知道 user 这个 ORM 对象是什么东西

# 手动写关键字参数——太麻烦
UserInfoResponse(id=user.id, username=user.username, nickname=user.nickname)
# 字段少还行，字段多了写死人
```

### 解决：from_attributes + model_validate

```python
from pydantic import BaseModel, ConfigDict

class UserInfoResponse(BaseModel):
    id: int
    username: str
    nickname: str | None = None

    # 打开开关：允许从对象的属性（点号）取值
    model_config = ConfigDict(from_attributes=True)

# 然后一行搞定
user_info = UserInfoResponse.model_validate(user)
# Pydantic 内部做了：
#   user_info.id = user.id                   ← user.id → int
#   user_info.username = user.username       ← user.username → str
#   user_info.nickname = user.nickname        ← user.nickname → str | None
#   (password、created_at 等字段自动忽略！)
```

### 完整流程：数据库 → ORM → Pydantic → JSON

```text
db.query(User).first()
    │
    ▼
ORM 对象（User 实例）           ← 有 password、created_at 等多余字段
    │ model_validate(user)
    ▼
Pydantic 模型（UserInfoResponse） ← 只保留 id, username, nickname
    │ jsonable_encoder()
    ▼
纯字典                          ← 全部是 JSON 原生类型
    │ JSONResponse(content=...)
    ▼
JSON 字符串 ← 发送给前端
```

### 关于 model_name.model_validate 的疑问

你可能会在代码中看到两种写法：

```python
# 写法一：Pydantic v2 官方推荐
UserInfoResponse.model_validate(user)

# 写法二：Pydantic v1 旧写法
UserInfoResponse.from_orm(user)
```

在 Pydantic v2 中，`model_validate()` 替代了 v1 的 `from_orm()`。两种写法现在都支持，但推荐用 `model_validate`。

## 六、ConfigDict 配置详解

`model_config = ConfigDict(...)` 是 Pydantic v2 的模型配置入口，控制整个模型的行为。

### 1. `from_attributes=True` （最常用）

**作用**：允许从对象的属性（`点号`）取数据，而不是只接受字典或关键字参数。

```python
# ❌ 没有 from_attributes
UserInfoResponse(user)                           # 报错！不知道 user 是什么
UserInfoResponse(id=user.id, username=user.username)  # 能行，但手动写

# ✅ 有 from_attributes
UserInfoResponse.model_validate(user)
# 自动：id = user.id, username = user.username, ...
```

**本质**：`model_validate(obj)` 等价于手动写关键字参数：

```python
# 手动（麻烦）
UserInfoResponse(id=user.id, username=user.username, nickname=user.nickname)

# 自动（一行）
UserInfoResponse.model_validate(user)
#       ↑ 遍历 schema 的所有字段，从 user 身上读同名字段
```

**适用场景**：路由里拿到的是 **ORM 对象**（SQLAlchemy），需要转成 Pydantic 再返回。

---

### 2. `populate_by_name=True` （配合 alias 使用）

**作用**：有 `alias` 别名的字段，既可以用别名赋值，也可以用 Python 原名赋值。

```python
# schema 定义
publish_time: datetime = Field(..., alias="publishTime")

# ❌ 没有 populate_by_name
NewsItemResponse(publish_time="...")   # 报错！只能写 publishTime
NewsItemResponse(publishTime="...")    # 能行，但代码里写驼峰很奇怪

# ✅ 有 populate_by_name
NewsItemResponse(publish_time="...")   # Python 风格，能行
NewsItemResponse(publishTime="...")    # 别名风格，也能行
```

**为什么需要**：两个地方用不同的名字：

| 场景 | 用哪个名字 | 示例 |
|---|---|---|
| `model_validate(obj)` 从 ORM 读属性 | **Python 名** | `obj.publish_time` |
| 手动构造模型（如 `UserAuthResponse(token=token, user_info=...)`） | **Python 名** | `UserAuthResponse(user_info=...)` |
| 序列化成 JSON 给前端 | **别名**（驼峰） | `{"publishTime": "..."}` |

**没有 `populate_by_name` 的问题**：

```python
# users.py
class UserAuthResponse(BaseModel):
    token: str
    user_info: UserInfoResponse = Field(..., alias="userInfo")

# router 里：
response_data = UserAuthResponse(token=token, user_info=user_info)
#                                          ↑ 用的是 Python 名 user_info
# 没有 populate_by_name → 报错，必须写成 userInfo
# 有 populate_by_name  → 正常，user_info 和 userInfo 都认
```

---

### 3. 两个配置的关系

```python
model_config = ConfigDict(
    populate_by_name=True,   # ① 构造时认 Python 名
    from_attributes=True,    # ② 允许从 ORM 对象读属性
)
```

- `from_attributes=True` — **必须和 `model_validate` 配合**，否则 `model_validate(obj)` 会报错
- `populate_by_name=True` — **必须和 `alias` 配合**，否则代码里只能用驼峰名构造
- 两者独立，可以只用其中一个

### 4. 何时用哪个

| 场景 | from_attributes | populate_by_name |
|---|---|---|
| ORM 对象 → Pydantic（最常用） | ✅ 要 | 建议加 |
| 纯字典/JSON → Pydantic | ❌ 不用 | 建议加 |
| 有 alias 的字段在代码里构造 | 按需 | ✅ 要 |
| 没有 alias 的字段 | 按需 | ❌ 不用 |

**本项目最佳实践**：所有响应模型都加两者，省心。

---

### 5. model_validate 时字段不匹配的规则

```python
user = User(id=1, username="jack", password="123456", phone="138xxx")

class UserOut(BaseModel):
    id: int
    username: str
    nickname: Optional[str] = None   # 有默认值
    # 没有 password, phone

out = UserOut.model_validate(user)
```

| 场景 | 结果 |
|---|---|
| ORM 属性多，schema 字段少（如 `password`、`phone`） | **自动忽略**，安全过滤 |
| ORM 属性少，schema 字段有默认值（如 `nickname`） | **用默认值填充**，不报错 |
| ORM 属性少，schema 字段无默认值（如 `id`） | **报 ValidationError**，字段缺失 |

所以可选字段建议都给默认值 `Optional[str] = None`，避免 ORM 查询漏列时崩掉。

## 七、整体调用链

```
路由函数：
    return success_response(data=response_data)

步骤分解：
    ① ORM 对象 → Pydantic 模型
       user_info = UserInfoResponse.model_validate(user)

    ② 组装成统一响应
       response_data = UserAuthResponse(token=token, user_info=user_info)

    ③ 传给 success_response
       content = {"code": 200, "message": "注册成功", "data": response_data}

    ④ jsonable_encoder 递归展平
       → {"code": 200, "message": "注册成功", "data": {"id": 1, "username": "jack", ...}}

    ⑤ JSONResponse 序列化 + 设置 Content-Type: application/json
       → 发送给客户端
```

> **💡 HTTP 知识关联**：步骤 ⑤ 中的 `Content-Type: application/json` 就是 `HTTP基础.md` 里讲的**响应头**。最终在网络上传输的原始报文长这样：
>
> ```text
> HTTP/1.1 200 OK                          ← 状态行（status_code 默认 200）
> Content-Type: application/json           ← JSONResponse 自动加的头
> Content-Length: 123
>                                          ← 空行
> {"code": 200, "message": "注册成功", ...} ← 响应体
> ```
>
> **注意这里有两个 200**：状态行的 `200 OK` 是 **HTTP 状态码**（告诉浏览器连接正常），body 里的 `"code": 200` 是**业务状态码**（告诉前端业务成功）。两者分开。详见 `HTTP基础.md` 第 10 节「这些概念在你项目里是怎么体现的」。

## 八、补充：response_model（FastAPI 路由装饰器）

```python
@router.post("/register", response_model=UserAuthResponse)
async def register(...):
    ...
```

`response_model` 是 **FastAPI 路由装饰器的参数**，作用是自动序列化 + 字段过滤：

```python
class UserOut(BaseModel):
    id: int
    username: str
    # 没有 password！

@router.get("/user/{id}", response_model=UserOut)
async def get_user(id: int):
    user = db.query(User).get(id)  # user 有 password 字段
    return user  # 直接返回 ORM，FastAPI 自动转
```

FastAPI 自动：
1. 用 `from_attributes=True` 读取属性
2. 只保留 `UserOut` 中声明的字段（password 被过滤）
3. 序列化成 JSON 返回

### response_model vs 手动 success_response

| | `response_model` | `success_response`（本项目） |
|---|---|---|
| 属于谁 | FastAPI | 自己写的工具函数 |
| 作用 | 自动序列化 + 过滤字段 | 统一包一层 `{code, message, data}` |
| 返回值 | Pydantic 模型 / ORM 对象 | 已组装好的 `JSONResponse` |

两者不冲突，可以同时使用。本项目选择手动控制（返回 `JSONResponse`），所以没写 `response_model`。

## 九、本项目完整示例

### schemas/users.py

```python
class UserInfoResponse(UserInfoBase):
    id: int
    username: str
    model_config = ConfigDict(from_attributes=True)


class UserAuthResponse(BaseModel):
    token: str
    user_info: UserInfoResponse = Field(..., alias="userInfo")
    model_config = ConfigDict(
        populate_by_name=True,
        from_attributes=True,
    )
```

### routers/users.py

```python
@router.post("/register")
async def register(user_data: UserRequest, db: AsyncSession = Depends(get_db)):
    existing_user = await users.get_user_by_username(db, user_data.username)
    if existing_user:
        raise HTTPException(status_code=400, detail="用户已存在")

    user = await users.create_user(db, user_data)
    token = await users.create_token(db, user.id)

    response_data = UserAuthResponse(
        token=token,
        user_info=UserInfoResponse.model_validate(user),
    )
    return success_response(message="注册成功", data=response_data)
```
