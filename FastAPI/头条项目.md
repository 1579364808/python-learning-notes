## 工程结构
### 二、 核心目录职责详解

为了实现“单一职责原则”和“高内聚低耦合”，各层级的分工非常明确：

#### 1. 入口点：`main.py`
只负责**初始化应用**和**全局配置**。包括：
* 创建 `FastAPI()` 实例。
* 注册全局 CORS 跨域中间件（允许 Vue 3 前端顺利调用接口）。
* 统一挂载注册子路由（例如：`app.include_router(news.router)`）。
* 配置全局异常处理器。

#### 2. 路由层：`routers/`
使用 FastAPI 提供的 `APIRouter`：
* **前缀与标签隔离**：例如新闻模块定义为 `router = APIRouter(prefix="/api/news", tags=["news"])`。
* **不直接写 SQL 或大段业务逻辑**：路由层函数主要负责**接收参数、调用 `crud/` 层的函数进行数据处理、最后返回响应**。
* **依赖注入**：通过 `Depends(get_db)` 动态注入数据库异步会话（Session）。

#### 3. 数据访问层：`crud/`
* 采用 Python 的异步协程操作。
* 编写底层的异步 SQLAlchemy 语法（如 `select()`、`insert()`、`update()` 和 `delete()`）。
* 将纯粹的数据存取行为与 API 路由隔离，使代码能复用。例如，在用户登录时需要查询用户，在更新个人信息时也需要查询用户，都调用 `crud/users.py` 里的同一个查询方法即可。

#### 4. 数据校验层：`schemas/`
* 基于 **Pydantic**。
* 解决两个痛点：**入参校验**（前端传来的数据格式对不对，如邮箱格式、密码长度）和**出参过滤**（数据库查询出的用户数据包含密码哈希值，通过设置 `response_model`，在输出时自动过滤掉敏感密码，只返回昵称和 ID）。

### 三、 课程中的高频“避坑点”

视频在讲解工程结构时，特意提到了学员极易遇到的几个经典问题：

1. **路由导入与循环引用（包加载机制问题）**：
   在 `main.py` 中挂载路由时，容易写成 `from routers import news`。
   * **报错原因**：由于 `main.py` 在包路径下启动，会导致 Python 无法在全局顶层搜寻到 `routers` 模块。
   * **解决方案**：使用相对导入 `from .routers import news`（即“从当前包下的 routers 子模块导入”）。
2. **异步 ORM 数据库文件与基类引用**：
   在设计 SQLAlchemy 模型时，各个模块的 Model 类（如 `User`, `News`）必须继承自同一个 `DeclarativeBase` 基类。如果在建表或生成表结构时引用不一致，会导致表无法在 MySQL 中被自动扫描创建。

## 模块化路由

在这一步中，我们专注于解决一个实际问题：**当你的项目越写越大，有几十甚至上百个接口时，如果全部写在一个文件中，代码会变得极其混乱、难以维护。**

为了解决这个问题，FastAPI 提供了 **`APIRouter`** 工具。我们将通过一个通俗易懂的比喻和具体的代码结构，分步学习如何进行**模块化路由拆分**。

---

### 第一步：理解核心概念 —— 什么是 `APIRouter`？

#### 💡 难懂概念拆解：`FastAPI` 与 `APIRouter` 的区别

* **`FastAPI` 实例（即 `app = FastAPI()`）**：
  * **比喻**：它就像是电脑的**主板**。一台电脑只能有一个主板，它负责提供电源、全局设置（比如跨域、全局中间件、文档生成）以及启动整个系统（Uvicorn 运行的就是它）。
* **`APIRouter` 实例（即 `router = APIRouter()`）**：
  * **比喻**：它就像是插在主板上的**扩展卡（显卡、声卡）**或 **USB 扩展坞**。
  * 扩展卡本身不能独立开机运行，它必须“插在”主板上才能工作。
  * 我们可以把“用户相关接口”做成一个“用户扩展卡”，把“新闻相关接口”做成一个“新闻扩展卡”，最后统一插到主板（`FastAPI`）上。

---

### 第二步：实战演练 —— 拆分“用户模块”路由

我们现在要在 `routers/users.py` 文件中，单独定义用户登录和注册的接口。

#### 1. 编写子路由文件：`routers/users.py`

在这个文件里，我们**不再**使用 `app = FastAPI()`，而是导入并使用 `APIRouter`：

```python
# routers/users.py
from fastapi import APIRouter

# 创建一个路由扩展坞
router = APIRouter(
    prefix="/api/users",  # 路由前缀（重要概念，下文会解释）
    tags=["用户模块"]      # Swagger 文档中的分组标签
)

# 定义注册接口：实际访问路径将是 /api/users/register
@router.post("/register")
async def register():
    return {"message": "注册成功"}

# 定义登录接口：实际访问路径将是 /api/users/login
@router.post("/login")
async def login():
    return {"message": "登录成功"}
```

#### 💡 难懂概念拆解：`prefix` 与 `tags`

* **`prefix="/api/users"`（路由前缀）**：
  * 它的作用是**自动拼接路径**。
  * 只要在这个 `router` 下定义的接口，FastAPI 都会自动在它们的前面加上 `/api/users`。
  * 比如你在代码里写的是 `@router.post("/login")`，但前端实际请求的完整路径是 `http://127.0.0.1:8000/api/users/login`。这样可以避免每个接口都重复写冗长的路径。
* **`tags=["用户模块"]`（接口标签）**：
  * 它是专门给 Swagger 自动生成文档（`/docs`）用的。
  * 加上这个参数后，在浏览器访问 `/docs` 时，登录和注册接口会被整齐地归类在“用户模块”折叠栏下方，不会和其他模块的接口混在一起。

---

### 第三步：挂载路由 —— 将“扩展卡”插上“主板”

定义好 `routers/users.py` 之后，直接启动 Uvicorn 是无法访问这些接口的，因为主板（`main.py`）还不知道它们的存在。我们需要进行“挂载”（即注册路由）。

#### 2. 编写主入口文件：`main.py`

```python
# main.py
from fastapi import FastAPI
# 导入我们刚刚写好的用户路由
from .routers.users import router as users_router

# 1. 初始化主板（主应用）
app = FastAPI(title="AI 掘金头条 API")

# 2. 将用户路由“插”到主应用上
app.include_router(users_router)

# 主应用自己的根测试路径
@app.get("/")
async def root():
    return {"status": "Backend is running!"}
```

#### 💡 难懂概念拆解：`app.include_router()`

* 它的作用就是把我们写好的 `users_router` 注册进全局的路由树中。
* 此时，FastAPI 会扫描 `users_router` 下的所有路径（如 `/register`, `/login`），并结合它的前缀 `/api/users`，正式向外暴露这两个 API 接口。

## 第一步：数据库设计与 ORM 实体类定义（基础骨架）

```markdown
用 Python SQLAlchemy 2.0 风格，根据下面的 DDL 生成模型代码：
1.
创建一个公共基类 Base(DeclarativeBase)，包含 created_at 和 updated_at 两个 DateTime 字段，分别记录创建和更新时间
2.
每张表对应一个模型类，继承 Base
3.
每个模型类加上 __repr__ 方法
```

在开发任何后端功能之前，首先要明确数据在数据库中是如何存储的，并通过 Python 代码将表结构映射为对象模型（即 ORM）。新闻模块核心涉及两张表：**新闻分类表** 和 **新闻表**。

### 1. 统一基类定义与时间审计字段

在 SQLAlchemy 2.0 中，推荐使用全新声明式基类（`DeclarativeBase`）写法。为了使每张表都自动带有“创建时间”和“更新时间”字段，我们需要先定义一个包含这些公共属性的基类：

```python
# models/base.py 或直接在 models/news.py 顶部定义
from datetime import datetime
from sqlalchemy import DateTime
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

class Base(DeclarativeBase):
    """
    所有 ORM 模型的统一基类
    统一提供创建时间（create_at）和更新时间（update_at）审计字段
    """
    create_at: Mapped[datetime] = mapped_column(
        DateTime, 
        default=datetime.now, 
        comment="创建时间"
    )
    update_at: Mapped[datetime] = mapped_column(
        DateTime, 
        default=datetime.now, 
        onupdate=datetime.now,  # 数据有更新时，SQLAlchemy 自动更新该时间
        comment="更新时间"
    )
```

### 2. 新闻分类表模型 (`Category`)

新闻分类对应今日头条上方的导航栏（如：推荐、科技、娱乐、体育等）。

*   **表名**：`news_category`
*   **核心字段设计**：

```python
# models/news.py

class Category(Base):
    """新闻分类表 ORM 模型"""
    __tablename__ = "news_category"  # 显式指定 MySQL 中的表名

    id: Mapped[int] = mapped_column(
        Integer, 
        primary_key=True, 
        autoincrement=True, 
        comment="分类ID"
    )
    name: Mapped[str] = mapped_column(
        String(50), 
        unique=True, 
        nullable=False, 
        comment="分类名称"
    )
    sort_order: Mapped[int] = mapped_column(
        Integer, 
        default=0, 
        nullable=False, 
        comment="排序（值越小越靠前）"
    )

    def __repr__(self):
        return f"<Category(id={self.id}, name={self.name}, sort_order={self.sort_order})>"
```

### 3. 新闻内容表模型 (`News`)

新闻内容表用于存放每一篇具体的新闻稿件，并通过外键（Foreign Key）与对应的分类进行关联。

*   **表名**：`news`
*   **核心字段设计**：

```python
# models/news.py

from sqlalchemy import Integer, String, Text, ForeignKey
from sqlalchemy.orm import relationship

class News(Base):
    """新闻表 ORM 模型"""
    __tablename__ = "news"

    id: Mapped[int] = mapped_column(
        Integer, 
        primary_key=True, 
        autoincrement=True, 
        comment="新闻ID"
    )
    title: Mapped[str] = mapped_column(
        String(100), 
        nullable=False, 
        comment="新闻标题"
    )
    content: Mapped[str] = mapped_column(
        Text, 
        nullable=False, 
        comment="新闻正文内容"
    )
    description: Mapped[str] = mapped_column(
        String(255), 
        nullable=True, 
        comment="新闻摘要/导语"
    )
    image: Mapped[str] = mapped_column(
        String(255), 
        nullable=True, 
        comment="新闻封面图 URL"
    )
    author: Mapped[str] = mapped_column(
        String(50), 
        nullable=False, 
        comment="作者"
    )
    views: Mapped[int] = mapped_column(
        Integer, 
        default=0, 
        nullable=False, 
        comment="浏览量/阅读数"
    )
    category_id: Mapped[int] = mapped_column(
        Integer, 
        ForeignKey("news_category.id"),  # 建立与 news_category 表的外键关系
        nullable=False, 
        comment="分类ID"
    )

    # 建立多对一的关系属性（可选，方便在查询 News 时直接获取对应的 Category 对象）
    category = relationship("Category", backref="news_list")

    def __repr__(self):
        return f"<News(id={self.id}, title={self.title}, views={self.views})>"
```

### 4. 笔记补充：SQLAlchemy 2.0 新特性“避坑”解析

在做这部分笔记时，你需要特别记录以下两点：
1.  **为什么写成 `Mapped[int]` 形式？**
    这是 SQLAlchemy 2.0 的新标准。在旧版（1.x）中，我们写 `id = Column(Integer, ...)`。新版通过 Python 类型注解（`Mapped[...]`）配合 `mapped_column()`，能让 VS Code 或 PyCharm **提供极佳的静态类型检查与代码补全**，大大降低了开发拼写错误率。
2.  **基类 Base 的唯一性：**
    所有在 `routers` 和 `crud` 里操作的模型都必须统一继承同一个 Base。如果分类表和新闻表不小心继承了两个在不同文件创建的基类，异步 ORM 引擎在执行 `create_all` 生成表结构时会报找不到关联表的错误。

## 第二步：Pydantic 请求与响应模型（Schemas）的设计

在定义好底层数据库的 ORM 骨架后，我们不能直接把数据库 Model 对象扔给前端，原因有两个：
1. **安全性与带宽浪费**：ORM 对象通常包含完整的数据库字段（甚至包括一些敏感信息或不需要的大文本字段，如列表页不需要传输几十 KB 的正文 `content`）。
2. **格式不匹配**：数据库的 `datetime` 字段返回给前端时是默认的 ISO 格式，前端不方便直接展示，需要统一格式化。

因此，我们需要使用 FastAPI 自带的 **Pydantic** 建立数据契约层（Schemas），负责**入参校验**和**出参过滤**。

---

### 1. 分类数据响应模型 (`CategoryOut`)

前端获取分类列表时，需要一个规范的 JSON 结构。

```python
# schemas/news.py
from pydantic import BaseModel, Field, ConfigDict

class CategoryOut(BaseModel):
    """新闻分类输出模型"""
    id: int = Field(..., description="分类ID")
    name: str = Field(..., description="分类名称")
```

## 第三步：异步数据访问层（CRUD）的具体实现

在 FastAPI 高并发架构中，传统的同步数据库查询（会阻塞线程）是不被推荐的。因此，《AI掘金头条》项目采用的是 **SQLAlchemy 2.0+ 的异步模式（`AsyncSession`）**。

这一步，我们编写 `crud/news.py`。该层不包含任何 HTTP 请求处理逻辑，只专注于**如何高效地从 MySQL 中读取和修改数据**。

### 1. 异步获取所有新闻分类列表

由于分类数据不多，直接查出并按照 `sort_order`（权重值越小越靠前）进行升序排序。

```python
# crud/news.py
from sqlalchemy import select, func
from sqlalchemy.ext.asyncio import AsyncSession
from models.news import Category, News
from schemas.news import NewsQueryParams

async def get_all_categories(db: AsyncSession):
    """
    异步获取所有新闻分类列表
    """
    # 构造查询语句，按 sort_order 升序排序
    stmt = select(Category).order_by(Category.sort_order.asc())
    
    # 异步执行并获取标量结果
    result = await db.execute(stmt)
    return result.scalars().all()
```

### 2. 异步获取分页新闻列表（带分类过滤）

这里需要实现两个核心逻辑：
1. **获取满足条件的新闻总数**（计算 `total`）。
2. **分页切片查询**（根据 `page` 和 `size` 计算 `offset` 偏移量进行查询）。

```python
async def get_news_list(db: AsyncSession, params: NewsQueryParams):
    """
    分页获取新闻列表
    """
    # 1. 构造基础查询语句（按创建时间降序排序，最新发布的新闻在最前面）
    stmt = select(News).order_by(News.create_at.desc())
    
    # 2. 如果前端传了分类ID，则添加过滤条件
    if params.category_id is not None:
        stmt = stmt.where(News.category_id == params.category_id)
        
    # 3. 异步查询符合条件的总记录数 (Total Count)
    count_stmt = select(func.count()).select_from(stmt.subquery())
    total_result = await db.execute(count_stmt)
    total = total_result.scalar() or 0
    
    # 4. 加上 limit 和 offset 进行物理分页
    offset_val = (params.page - 1) * params.size
    stmt = stmt.offset(offset_val).limit(params.size)
    
    # 5. 异步执行查询，获取当前页的数据
    news_result = await db.execute(stmt)
    items = news_result.scalars().all()
    
    # 6. 计算是否还有下一页
    has_more = total > (params.page * params.size)
    
    return total, items, has_more
```

### 3. 获取新闻详情并自动累加浏览量（Views）

为了避免高并发下多表查询的复杂性，这里在查询新闻详情的同时，对该文章的 `views` 字段执行 `+1` 操作，并提交到数据库。

```python
async def get_news_detail_and_update_views(db: AsyncSession, news_id: int):
    """
    根据新闻ID获取详情，并更新该新闻的浏览量
    """
    # 1. 查询新闻是否存在
    stmt = select(News).where(News.id == news_id)
    result = await db.execute(stmt)
    news_item = result.scalar_one_or_none()
    
    if news_item:
        # 2. 如果存在，浏览量自增 1
        news_item.views += 1
        
        # 3. 提交事务，写入数据库
        await db.commit()
        # 4. 刷新内存对象，确保拿到最新的 views 值和更新时间
        await db.refresh(news_item)
        
    return news_item
```

### 4. 笔记补充：异步 ORM 核心“避坑”解析

在做本节笔记时，你需要记录以下异步数据库操作的要点：
1. **为什么用 `await db.execute()`？**
   在同步的 SQLAlchemy 中，我们可以使用 `db.query(News)`。但在异步模式下，必须配合 Python 的 `await` 关键字，显式利用 `select(News)` 构建语句并由 `db.execute()` 或 `db.scalars()` 异步执行，否则会产生线程阻塞。
2. **`scalar_one_or_none()` 与 `scalar()` 的区别**：
   * `scalar_one_or_none()` 用于查询单条记录，如果有且仅有一条则返回，没有则返回 `None`；如果查询出多条会抛出异常，非常适合按主键 ID 查询。
   * `scalar()` 适合返回聚合函数（如 `func.count()`）的单一数值。
3. **性能隐患（浏览量实时写入问题）**：
   *在真实的大流量头条项目中，每次用户点击详情都执行 `db.commit()` 会对 MySQL 产生很大的写压力。课程在进阶部分会提到，在面对高并发时，更优的方案是**将浏览量先写入 Redis 缓存**，再通过定时任务（或异步消费者）批量同步回 MySQL。*

### 1. 数据库异步连接依赖注入准备

在开始编写路由前，先了解 `get_db` 依赖函数是如何工作的。它定义在 `config/db_config.py` 中，主要作用是**在每个 HTTP 请求到达时自动开启一个异步数据库 Session，并在请求结束（无论成功或异常）时自动关闭该 Session**，避免连接泄露：

```python
# config/db_config.py 简化参考代码
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession

# 异步引擎配置（使用 aiomysql 驱动）
DATABASE_URL = "mysql+aiomysql://root:password@localhost:3306/news_app"
engine = create_async_engine(DATABASE_URL, echo=False)
AsyncSessionLocal = async_sessionmaker(bind=engine, expire_on_commit=False)

async def get_db():
    """依赖注入生成器：提供异步数据库 Session"""
    async with AsyncSessionLocal() as session:
        yield session
        # 请求结束后，在此处自动执行上下文清理，释放 Session
```

### 5. 笔记补充：API 路由层核心“避坑”解析

在做本节笔记时，请务必标注关于 **接口健壮性** 与 **FastAPI 特性** 的考点：
1. **依赖注入中 `Depends()` 的两种传参形式**：
   * `db: AsyncSession = Depends(get_db)`：表示执行 `get_db` 异步函数获取生成值并注入。
   * `params: NewsQueryParams = Depends()`：当 Pydantic 模型作为 GET 请求的 Query 参数时，如果不写 `Depends()`，FastAPI 默认会去 **Request Body** 中解析 JSON 数据（通常 POST 请求才这样做）。写上 `Depends()` 后，FastAPI 就会聪明地将 `?category_id=1&page=2&size=10` 等 URL 查询参数绑定到 Pydantic 对象中。
2. **`response_model` 的多重功效**：
   * 告诉 Swagger 交互文档这个接口会返回什么格式。
   * **动态过滤数据**：例如我们的 `News` 数据库实体包含大文本 `content`，但在 `get_news_list` 接口中，我们的 `response_model` 是 `PageResult[NewsBriefOut]`。由于 `NewsBriefOut` 里面没有定义 `content` 字段，FastAPI 在输出时会自动过滤、剥离掉 `content` 数据，从而极大节省了网络传输带宽。

## 第五步：高性能缓存优化 —— Redis 与新闻模块的高并发集成

在真实的资讯类应用（如今日头条）中，新闻分类、首页新闻列表、热门新闻详情等数据属于**高频访问、低频修改**的热点数据。如果每次页面刷新或滑动都直接去 MySQL 查询，数据库连接池很容易被瞬间占满，导致系统瘫痪。

因此，本套教程在实战阶段引入了 **Redis 异步缓存机制**（基于 `redis.asyncio`），对新闻模块的数据读取进行了大幅度并发优化。

---

### 1. 异步 Redis 客户端配置

在 `config/cache_conf.py` 中，采用异步模式连接 Redis，并封装通用的“读取”与“写入”辅助函数：

```python
# config/cache_conf.py
import json
from typing import Any, Optional
import redis.asyncio as redis

REDIS_HOST = "localhost"
REDIS_PORT = 6379
REDIS_DB = 2
REDIS_PWD = "your_password"

# 创建全局唯一的异步 Redis 连接对象
redis_client = redis.Redis(
    host=REDIS_HOST,
    port=REDIS_PORT,
    db=REDIS_DB,
    password=REDIS_PWD,
    decode_responses=True  # 自动将字节数据（bytes）解码为 Python 字符串（str）
)

async def get_json_cache(key: str) -> Optional[Any]:
    """读取并解析 JSON 缓存数据"""
    try:
        data = await redis_client.get(key)
        if data:
            return json.loads(data)
        return None
    except Exception as e:
        print(f"读取 Redis 缓存失败: {e}")
        return None

async def set_json_cache(key: str, value: Any, expire: int = 3600) -> bool:
    """序列化并保存数据为 JSON 缓存"""
    try:
        if isinstance(value, (dict, list)):
            value = json.dumps(value, ensure_ascii=False)
        await redis_client.setex(key, expire, value)
        return True
    except Exception as e:
        print(f"写入 Redis 缓存失败: {e}")
        return False
```

### 2. 核心技术痛点：ORM 对象如何缓存至 Redis？

数据库查出来的原始数据是 SQLAlchemy 的 **ORM 实体对象**，它们包含特殊的底层状态，**不能直接用 `json.dumps()` 转换**。
课程中给出了一个非常巧妙且符合 Pydantic v2 标准的解决方案：

*   **转换公式**：`ORM 对象列表` -> `Pydantic 模型实例化（数据清洗）` -> `Python 基础字典列表（可序列化）` -> `Redis JSON 字符串`

**转换函数示例**：
```python
# utils/cache_helper.py
from typing import List
from pydantic import ConfigDict
from schemas.news import NewsBriefOut

def serialize_news_list(news_orm_list) -> List[dict]:
    """
    将 SQLAlchemy 的 ORM 列表转换成能转为 JSON 的字典列表
    """
    # 借助 Pydantic v2 的 model_validate 转换数据
    return [
        NewsBriefOut.model_validate(item).model_dump(mode="json")
        for item in news_orm_list
    ]
```
*(注意：此处 `NewsBriefOut` 必须配置有 `model_config = ConfigDict(from_attributes=True)`，才能顺利读取 ORM 的属性)*。

### 3. “旁路缓存（Cache Aside Pattern）”读取机制的引入

以获取新闻列表为例，我们在 API 路由（`routers/news.py`）中应用**旁路缓存策略**：
*   **第 1 步**：先去 Redis 查询是否有当前分类、页码、大小的缓存。
*   **第 2 步（Cache Hit）**：如果缓存命中，直接反序列化为 JSON 返回给前端，**无需查询 MySQL**。
*   **第 3 步（Cache Miss）**：如果缓存未命中，再去 MySQL 中执行查询，然后将查询结果转换并写入 Redis，设定生存时间（TTL），最后返回。

```python
# routers/news.py 中的优化实现
from config.cache_conf import get_json_cache, set_json_cache
from utils.cache_helper import serialize_news_list

@router.get("/list", response_model=PageResult[NewsBriefOut])
async def get_news_list_optimized(
    params: NewsQueryParams = Depends(),
    db: AsyncSession = Depends(get_db)
):
    # 1. 动态拼接 Redis 唯一的 Key
    # 例如：news:list:1:2:10（分类为1，第2页，每页10条）
    category_flag = params.category_id if params.category_id is not None else "all"
    cache_key = f"news:list:{category_flag}:{params.page}:{params.size}"
    
    # 2. 尝试从 Redis 中获取数据
    cached_data = await get_json_cache(cache_key)
    if cached_data:
        # 缓存命中，直接返回
        return cached_data

    # 3. 缓存未命中，走数据库查询
    total, items, has_more = await news_crud.get_news_list(db, params)
    
    # 4. 将数据库 ORM 结果序列化为可转 JSON 的原生字典格式
    serialized_items = serialize_news_list(items)
    response_data = {
        "total": total,
        "items": serialized_items,
        "has_more": has_more
    }
    
    # 5. 将结果异步存入 Redis，设置缓存有效期为 1800 秒（30分钟）
    await set_json_cache(cache_key, response_data, expire=1800)
    
    return response_data
```

### 4. 缓存失效与更新机制（Cache Invalidation）

在课程的进阶章节中，视频特意强调了“数据一致性”的问题。如果后台编辑人员修改了某条新闻，或者用户删除了某条新闻，而 Redis 里的缓存依然生效，前端就会读取到“脏数据”。

*   **课程解决策略**：
    1.  **缓存失效（Lazy Delete）**：在对新闻执行 `UPDATE`（修改）、`DELETE`（删除）或添加新文章时，代码必须主动执行 `await redis_client.delete(cache_key)` 将相关缓存键清除，逼迫下一次查询重新请求 MySQL 并重建缓存。
    2.  **设置合理的过期时间（TTL）**：
        *   分类数据（变化极低）：设置缓存 2 小时（`news:categories`）。
        *   新闻列表页（更新中等）：设置缓存 30 分钟（`news:list:...`）。
        *   具体文章详情（高频访问）：设置缓存 1 小时（`news:detail:{news_id}`）。

### 5. 笔记补充：缓存优化核心“避坑”解析

在做这部分笔记时，请注意记录以下两点高频面试/开发考点：
1.  **缓存击穿（Hotspot Key）**：
    当一个极度热门的新闻详情（例如微博热搜第一名）在 Redis 中的缓存瞬间过期，而此时有上万个并发请求同时涌入。如果代码不加限制，这上万个请求会同时透传到 MySQL 数据库，导致 MySQL 宕机。在企业级开发中，需要结合分布式锁（`Redisson` 或 Redis 锁）或者设置“缓存永不过期，后台异步刷新”来避免。
2.  **`decode_responses=True` 的必要性**：
    如果不设置此参数，`redis.get()` 返回的将是类似于 `b'{"total": 100}'` 这样的 bytes 字节流，直接将其放入 `json.loads()` 会在一些旧版本的 Python 解释器中产生报错。显式配置该选项能极大地提高开发便利性。

---

通过引入 **Redis 异步旁路缓存**，新闻模块的数据读取性能相比直接读 MySQL 能提升数倍甚至数十倍，这也是大型 Web 系统的标准架构实践。

# 用户模块 - 第一步：用户数据库设计与密码加盐哈希

安全是用户系统的生命线。设计用户表时，最核心的原则就是：**绝对不能在数据库中明文存储用户密码**。项目中采用的是业界通用的 **`bcrypt` 算法进行密码加盐哈希**。

### 1. 用户表 ORM 模型定义 (`User`)

该模型定义在 `models/users.py` 中，继承自我们之前定义的 `Base` 基类（已包含 `create_at` 和 `update_at` 审计字段）。

*   **表名**：`news_user`
*   **字段设计**：

```python
# models/users.py
from sqlalchemy import Integer, String, Boolean
from sqlalchemy.orm import Mapped, mapped_column
from .base import Base  # 引入此前定义的统一基类

class User(Base):
    """系统用户表 ORM 模型"""
    __tablename__ = "news_user"

        Integer, 
        primary_key=True,

        comment="用户ID"
    )
    username: Mapped[str] = mapped_column(

        unique=True,

        nullable=False, 
        comment="登录账号/用户名"
    )
    hashed_password: Mapped[str] = mapped_column(
        String(128), 
        nullable=False, 
        comment="加密后的密码哈希值"
    )
    nickname: Mapped[str] = mapped_column(
        String(50), 
        nullable=True, 
        comment="用户昵称"
    )
    avatar: Mapped[str] = mapped_column(

        nullable=True,

        Boolean, 
        default=True, 
        nullable=False, 
        comment="账号是否激活/启用（可用于封禁机制）"
    )

    def __repr__(self):
        return f"<User(id={self.id}, username={self.username}, nickname={self.nickname})>"
```

### 2. 密码安全加盐哈希工具类开发

由于用户注册时输入的是明文密码，登录时输入的也是明文密码，我们需要在公共工具类 `utils/security.py` 中，使用 Python 的 `passlib` 库（底层调用 `bcrypt` 算法）来实现密码的**单向加密哈希**和**比对验证**。

在运行此部分前，通常需要安装依赖：
```bash
pip install "passlib[bcrypt]"
```

在 `utils/security.py` 中编写如下工具函数：

```python
# utils/security.py
from passlib.context import CryptContext

# 创建密码加密上下文，指定使用 bcrypt 算法进行哈希
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def get_password_hash(password: str) -> str:
    """
    1. 将用户输入的明文密码进行单向哈希加密（自动生成随机盐）
    """
    return pwd_context.hash(password)

def verify_password(plain_password: str, hashed_password: str) -> bool:
    """
    2. 比对明文密码与数据库中的哈希密码是否一致
    """
    return pwd_context.verify(plain_password, hashed_password)
```

---

### 3. 笔记补充：密码哈希核心“避坑”解析

在记录本部分笔记时，需留意以下安全与技术细节：
1.  **为什么不使用简单的 MD5 算法？**
    MD5 加密因为没有动态加盐，容易受到“彩虹表（Rainbow Tables）”的暴力破解攻击。而 `bcrypt` 算法在加密过程中，会自动为每一个密码生成一个**随机盐（Salt）**，即使两个用户设置了完全相同的密码（如 `123456`），加密后存入数据库的 `hashed_password` 也是完全不同的。此外，`bcrypt` 可以调整计算强度，防暴力破解能力强。
2.  **`username` 字段添加 `index=True` 索引**：
    在登录时，后端需要通过 `SELECT * FROM news_user WHERE username = :username` 来检索用户是否存在。在海量用户的情况下，为 `username` 加上单列索引可以极大地提升检索效率，减少登录时的响应时间。

---

这便是用户模块的底层数据设计和密码安全保障逻辑。

### 2. JWT 签发工具函数开发 (`utils/security.py`)

为了生成能够携带用户身份（如用户 ID、用户名）且防篡改的加密凭证，我们需要使用 `PyJWT` 库。

在 `utils/security.py` 中，定义签发 JWT Token 的核心参数和函数：

```python
# utils/security.py （续上一步代码）
import jwt
from datetime import datetime, timedelta, timezone
from typing import Optional

# 秘钥与加密算法配置（生产环境应配置到 .env 或环境变量中）
SECRET_KEY = "heima-toutiao-ai-secret-key-2026"
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 60 * 24  # 默认设置 Token 有效期为 24 小时

def create_access_token(data: dict, expires_delta: Optional[timedelta] = None) -> str:
    """
    根据传入的用户数据载荷（Payload）生成 JWT Token
    """
    to_encode = data.copy()
    
    # 1. 计算过期时间
    if expires_delta:
        expire = datetime.now(timezone.utc) + expires_delta
    else:
        expire = datetime.now(timezone.utc) + timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
    
    # 2. 注入过期时间（标准字段 exp）
    to_encode.update({"exp": expire})
    
    # 3. 使用 SECRET_KEY 与 HS256 算法进行签名加密
    encoded_jwt = jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt
```

### 3. 数据访问层（CRUD）编写 (`crud/users.py`)

实现**根据用户名查询用户**和**新用户入库**两个基础逻辑：

```python
# crud/users.py
from typing import Optional
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession
from models.users import User
from schemas.users import UserRegister
from utils.security import get_password_hash

async def get_user_by_username(db: AsyncSession, username: str) -> Optional[User]:
    """根据用户名查询用户（用于校验重名和登录核对）"""
    stmt = select(User).where(User.username == username)
    result = await db.execute(stmt)
    return result.scalar_one_or_none()

async def create_user(db: AsyncSession, user_in: UserRegister) -> User:
    """新用户写入数据库（包含明文密码哈希转换）"""
    hashed_pwd = get_password_hash(user_in.password)
    db_user = User(
        username=user_in.username,
        hashed_password=hashed_pwd,
        nickname=user_in.nickname or user_in.username,  # 若未传昵称，默认使用用户名
    )
    db.add(db_user)
    await db.commit()
    await db.refresh(db_user)
    return db_user
```

### 4. 路由控制层（Router）对接 (`routers/users.py`)

最后，我们在 `routers/users.py` 中暴露注册和登录两个路由端点：

```python
# routers/users.py
from fastapi import APIRouter, Depends, HTTPException, status
from sqlalchemy.ext.asyncio import AsyncSession

from config.db_config import get_db
from schemas.users import UserRegister, UserLogin, UserOut, Token
from crud import users as user_crud
from utils.security import verify_password, create_access_token

router = APIRouter(prefix="/api/users", tags=["用户账号模块"])


@router.post("/register", response_model=UserOut, status_code=status.HTTP_201_CREATED)
async def register(user_in: UserRegister, db: AsyncSession = Depends(get_db)):
    """
    用户注册接口
    """
    # 1. 校验用户名是否已注册
    db_user = await user_crud.get_user_by_username(db, user_in.username)
    if db_user:
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail="该用户名已被占用，请尝试其他用户名"
        )
    # 2. 写入数据库并返回公开的用户信息
    new_user = await user_crud.create_user(db, user_in)
    return new_user


@router.post("/login", response_model=Token)
async def login(user_in: UserLogin, db: AsyncSession = Depends(get_db)):
    """
    用户登录接口（成功则返回 Token）
    """
    # 1. 检索用户是否存在
    db_user = await user_crud.get_user_by_username(db, user_in.username)
    if not db_user:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="用户名或密码错误"
        )
    
    # 2. 比对用户输入的密码哈希是否匹配
    if not verify_password(user_in.password, db_user.hashed_password):
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="用户名或密码错误"
        )
        
    # 3. 生成包含用户核心身份（如用户ID）的 JWT Token 载荷
    token_data = {"sub": str(db_user.id), "username": db_user.username}
    access_token = create_access_token(data=token_data)
    
    # 4. 返回 Token 给前端保存
    return {
        "access_token": access_token,
        "token_type": "bearer"
    }
```

### 5. 笔记补充：路由层核心“避坑”解析

1.  **统一的错误信息提示**：
    在登录接口中，无论是找不到用户名，还是密码不匹配，后端统一向前端抛出 `"用户名或密码错误"` 的通用提示，而不是精准抛出 `"密码错误"` 或 `"用户名不存在"`。**这是行业标准的防黑客枚举攻击策略**。
2.  **JWT 载荷 `sub` 标准字段的运用**：
    按照 JWT 标准规范，`sub`（Subject）通常被指定为该 Token 主体的唯一 ID（此处放用户 ID）。因为数据库中的 ID 可能是 `int` 类型，而在 JWT 传输和校验中，我们常采用其字符串形式，便于解析和多语言兼容。

# 用户模块 - 第三步：高频核心，接口权限卫哨（OAuth2Bearer 与路由拦截）

在前后端分离的项目中，凡是涉及“获取个人信息”、“发布评论”、“点赞收藏”等操作的接口，后端都必须对请求进行拦截，判断该请求是否携带了有效的 JWT Token，进而识别是哪位用户在操作。

FastAPI 针对此场景提供了极其优雅且工程化的设计。在这一步，我们将编写系统的 **安全依赖注入函数 `get_current_user`**，作为整个后端的“路由卫哨”。

### 1. 引入并定义 OAuth2 安全机制

FastAPI 内置了符合 OAuth2 标准的 Token 提取机制。我们在 `utils/security.py` 或单独的 `dependencies/auth.py` 中，定义凭证提取规则：

```python
# utils/security.py (继续扩展)
from fastapi.security import OAuth2PasswordBearer
from fastapi import Depends, HTTPException, status
from sqlalchemy.ext.asyncio import AsyncSession
import jwt

from config.db_config import get_db
from crud import users as user_crud
from models.users import User

# 1. 实例化 OAuth2 Bearer 凭证提取器
# tokenUrl 指定登录接口的地址，配置后，Swagger 交互文档右上角会自动出现 "Authorize" 锁头按钮
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/api/users/login")
```

### 2. 编写全局核心权限依赖 `get_current_user`

这是一个**复合型依赖函数**。它不仅依赖于 `oauth2_scheme`（自动从 HTTP 请求头 `Authorization: Bearer <token>` 中提取 Token），还依赖于 `get_db`（获取数据库连接以便查询当前用户是否存在）。

```python
# utils/security.py (继续扩展)

async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db)
) -> User:
    """
    权限卫哨：解析 Token，提取当前登录用户
    """
    # 统一的非授权异常对象
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="您的登录凭证已失效或不合法，请重新登录",
        headers={"WWW-Authenticate": "Bearer"},
    )
    
    try:
        # 1. 解密并校验 Token 签名与过期时间（PyJWT 会自动处理签名和 exp 过期校验）
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id: str = payload.get("sub")
        
        if user_id is None:
            raise credentials_exception
            
    except jwt.ExpiredSignatureError:
        # Token 签名已经过期
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="登录已过期，请重新登录"
        )
    except jwt.InvalidTokenError:
        # Token 非法/被篡改
        raise credentials_exception

    # 2. 从数据库中查询该用户是否真实存在、是否仍处于激活状态
    user = await user_crud.get_user_by_username(db, payload.get("username"))
    # 或者写一个更高效的按 ID 查询: await user_crud.get_user_by_id(db, int(user_id))
    
    if user is None:
        raise credentials_exception
        
    if not user.is_active:
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN,
            detail="该账号已被封禁，无法操作"
        )
        
    # 3. 校验完全通过，将该 ORM 用户对象返回（它将被动态注入到有需要的路由函数中）
    return user
```

# 联姻篇 - 第一步：多对多中间表设计与索引优化

用户和新闻是典型的多对多关系：一个用户可以收藏多篇新闻，一篇新闻也可以被多个用户收藏；同理，浏览历史亦然。因此，我们需要创建两个独立的关联表：`favorite`（收藏表）与 `history`（浏览历史表）。

为了保证海量数据下的查询性能，我们在设计 SQLAlchemy 模型时，必须引入**联合唯一索引**与**单列高频检索索引**。

### 1. 收藏表模型定义 (`Favorite`)

用户在点击某篇文章时，系统需要高频判断“当前用户是否已收藏该文章”（用于前端高亮展示星星图标）。

*   **表名**：`favorite`
*   **设计要点**：
    *   `user_id` 和 `news_id` 分别是外键，指向各自的主表。
    *   设置 **联合唯一索引（Unique Constraint）**：同一位用户不可能对同一篇文章收藏两次。

```python
# models/favorite.py
from sqlalchemy import Integer, ForeignKey, Index
from sqlalchemy.orm import Mapped, mapped_column
from .base import Base  # 引入通用基类

class Favorite(Base):
    """用户收藏表 ORM 模型"""
    __tablename__ = "favorite"

    # 索引优化：创建联合唯一索引，确保 (user_id, news_id) 组合的唯一性，加快查询速度
    __table_args__ = (
        Index("uq_user_news_fav", "user_id", "news_id", unique=True),
    )

    id: Mapped[int] = mapped_column(
        Integer, 
        primary_key=True, 
        autoincrement=True, 
        comment="收藏自增ID"
    )
    user_id: Mapped[int] = mapped_column(
        Integer, 
        ForeignKey("news_user.id", ondelete="CASCADE"),  # 级联删除：用户被删除时，其收藏关联自动删除
        nullable=False, 
        comment="用户ID"
    )
    news_id: Mapped[int] = mapped_column(
        Integer, 
        ForeignKey("news.id", ondelete="CASCADE"),       # 级联删除：新闻被删除时，相关收藏记录自动清除
        nullable=False, 
        comment="新闻ID"
    )

    def __repr__(self):
        return f"<Favorite(user_id={self.user_id}, news_id={self.news_id})>"
```

### 2. 浏览历史表模型定义 (`History`)

浏览历史与收藏不同，它记录的是用户**阅读新闻的轨迹**。
*   **特性**：同一个用户可以多次重复阅读同一篇新闻，每次阅读都会刷新或增加历史足迹。因此，该表不需要建立联合唯一限制。
*   **表名**：`history`
*   **性能要求**：用户查看自己的“历史足迹”时，必定是**按照浏览时间倒序排列（最新浏览的在最上面）**。所以，必须对“浏览时间”字段建立高效索引。

```python
# models/history.py
from datetime import datetime
from sqlalchemy import Integer, DateTime, ForeignKey, Index
from sqlalchemy.orm import Mapped, mapped_column
from .base import Base

class History(Base):
    """用户浏览历史表 ORM 模型"""
    __tablename__ = "history"

    # 索引优化：
    # 1. 对 user_id 建立索引，方便快速拉取某位用户的所有历史足迹
    # 2. 对 view_time 建立索引，保障分页倒序（Order By view_time DESC）时的查询性能
    __table_args__ = (
        Index("idx_history_user_id", "user_id"),
        Index("idx_history_view_time", "view_time"),
    )

    id: Mapped[int] = mapped_column(
        Integer, 
        primary_key=True, 
        autoincrement=True, 
        comment="历史记录自增ID"
    )
    user_id: Mapped[int] = mapped_column(
        Integer, 
        ForeignKey("news_user.id", ondelete="CASCADE"), 
        nullable=False, 
        comment="用户ID"
    )
    news_id: Mapped[int] = mapped_column(
        Integer, 
        ForeignKey("news.id", ondelete="CASCADE"), 
        nullable=False, 
        comment="新闻ID"
    )
    view_time: Mapped[datetime] = mapped_column(
        DateTime, 
        default=datetime.now, 
        onupdate=datetime.now,  # 用户重复阅读同一篇文章时，自动更新浏览时间至最新
        nullable=False, 
        comment="最后一次浏览时间"
    )

    def __repr__(self):
        return f"<History(user_id={self.user_id}, news_id={self.news_id}, view_time={self.view_time})>"
```

### 3. 笔记补充：物理设计与外键级联“避坑”解析

在做本节笔记时，需留意这两个高频工程实践细节：
1.  **为什么显式定义 `ondelete="CASCADE"`？**
    如果不加级联删除（CASCADE），当管理员在后台删除了某一篇文章时，MySQL 默认会因为外键约束报错，导致新闻无法被成功删除；或者被迫先去手动清空该新闻的所有收藏/历史记录。
    加上级联删除后，主表记录一旦消亡，关联表的中间关系由 MySQL 数据库引擎在底层一并彻底清理，保证了数据一致性。
2.  **联合唯一索引 `uq_user_news_fav` 的双重作用**：
    *   **业务防呆**：从数据库底层彻底杜绝“由于前端并发多次点击或网络延迟造成的重复收藏记录”脏数据产生。
    *   **查询提速**：当我们在进入详情页检查“当前用户是否已收藏此新闻”时，数据库可以利用该索引，直接使用高效的复合索引扫描定位，不需要进行全表检索。

### 2. 异步数据访问层实现 (`crud/favorite.py`)

在这一层，利用 SQLAlchemy 2.0 的异步方法编写增删改查。特别注意：**在拉取用户收藏列表时，我们需要将 `Favorite` 表和 `News` 表进行内连接（INNER JOIN）**。

```python
# crud/favorite.py
from sqlalchemy import select, delete, func
from sqlalchemy.ext.asyncio import AsyncSession
from models.favorite import Favorite
from models.news import News

async def check_is_favorited(db: AsyncSession, user_id: int, news_id: int) -> bool:
    """检查用户是否收藏了某篇新闻"""
    stmt = select(Favorite).where(
        Favorite.user_id == user_id,
        Favorite.news_id == news_id
    )
    result = await db.execute(stmt)
    return result.scalar_one_or_none() is not None


async def add_news_favorite(db: AsyncSession, user_id: int, news_id: int) -> Favorite:
    """添加收藏"""
    # 如果已经收藏过，则不再重复插入
    stmt = select(Favorite).where(Favorite.user_id == user_id, Favorite.news_id == news_id)
    result = await db.execute(stmt)
    existing_fav = result.scalar_one_or_none()
    
    if existing_fav:
        return existing_fav
        
    db_fav = Favorite(user_id=user_id, news_id=news_id)
    db.add(db_fav)
    await db.commit()
    await db.refresh(db_fav)
    return db_fav


async def cancel_news_favorite(db: AsyncSession, user_id: int, news_id: int) -> bool:
    """取消收藏"""
    stmt = delete(Favorite).where(
        Favorite.user_id == user_id,
        Favorite.news_id == news_id
    )
    result = await db.execute(stmt)
    await db.commit()
    # rowcount 返回被删除的行数，若大于 0 说明删除成功
    return result.rowcount > 0


async def get_user_favorite_list(db: AsyncSession, user_id: int, page: int, size: int):
    """
    分页拉取用户的收藏新闻列表
    需要连表查询：Favorite INNER JOIN News ON Favorite.news_id = News.id
    """
    # 1. 构造基础连表语句，按收藏时间（favorite表的create_at）倒序排列
    stmt = (
        select(News)
        .join(Favorite, Favorite.news_id == News.id)
        .where(Favorite.user_id == user_id)
        .order_by(Favorite.create_at.desc())
    )
    
    # 2. 查询总条数
    count_stmt = select(func.count()).select_from(stmt.subquery())
    total_result = await db.execute(count_stmt)
    total = total_result.scalar() or 0
    
    # 3. 物理切片分页
    offset_val = (page - 1) * size
    stmt = stmt.offset(offset_val).limit(size)
    
    # 4. 执行查询，拿到 News 的 ORM 对象列表
    result = await db.execute(stmt)
    items = result.scalars().all()
    
    has_more = total > (page * size)
    return total, items, has_more
```

### 4. 笔记补充：多表 Join 查询核心“避坑”解析

在做这部分笔记时，需留意以下高频技术要点：
1.  **为什么不使用慢查询的 Python 循环？**
    初学者在获取“收藏的新闻列表”时，经常会写出如下代码：先去查 `Favorite` 表拿到所有的 `news_id` 列表，再写个循环去 `News` 表里按个 `get(news_id)`。
    *   **问题**：这产生了臭名昭著的 **N+1 次查询问题**，如果用户收藏了 100 篇文章，会向 MySQL 发送 101 次 SQL 请求，网络 IO 极其昂贵。
    *   **解决**：直接使用 SQL 关键字 `join(Favorite, Favorite.news_id == News.id)`。一次查询，数据库引擎内部自动匹配并返回新闻结果，大幅降低开销。
2.  **`response_model` 在多表关联下的复用**：
    在 `get_my_favorites` 路由中，虽然我们进行了多表联查，但我们的响应格式依然可以直接声明为 `response_model=PageResult[NewsBriefOut]`。
    因为连表 SQL 投影返回的直接就是 `News` 的 ORM 对象，Pydantic 会自动忽略 `Favorite` 的字段，直接将连表带出的 `News` 实体数据格式化并输出。

# 联姻篇 - 第三步：浏览历史（History）记录的业务与路由设计

在《AI掘金头条》中，**浏览历史模块**与收藏模块的开发有些相似，但它有一个非常重要的业务特点：
*   **高频刷新（Upsert 逻辑）**：同一个用户可以重复阅读同一篇新闻。如果用户今天读了《科技A》并产生了历史记录，明天又点开了《科技A》，系统不应该在数据库中新增一条完全重复的历史记录。
*   **正确做法**：先检查记录是否存在。若存在，则**更新（Update）该记录的浏览时间为当前时间**，让这条记录在历史列表中重新“置顶”；若不存在，再**创建（Insert）**新的历史记录。这被称为 **Upsert 逻辑**。

同时，为了适应前端开发规范（JavaScript 常用驼峰命名法 `camelCase`，而 Python 常用蛇形命名法 `snake_case`），课程在这里引入了 Pydantic v2 的 **别名（Alias）映射机制**。

---

### 1. Pydantic 别名机制与响应校验模型 (`schemas/history.py`)

为了将后端的 `view_time`、`news_id` 自动转换成前端需要的 `viewTime`、`newsId`，我们使用 `Field(..., alias="...")`：

```python
# schemas/history.py
from datetime import datetime
from typing import Optional, List
from pydantic import BaseModel, Field, ConfigDict

class HistoryAddRequest(BaseModel):
    """添加历史记录请求体"""
    news_id: int = Field(..., alias="newsId")

class HistoryItemResponse(BaseModel):
    """历史记录单项返回（内嵌新闻基本信息与浏览时间）"""
    id: int = Field(..., description="新闻ID")
    title: str = Field(..., description="新闻标题")
    description: Optional[str] = Field(None, description="文章导语")
    image: Optional[str] = Field(None, description="文章封面图")
    author: str = Field(..., description="作者")
    category_id: int = Field(..., alias="categoryId", description="分类ID")
    views: int = Field(..., description="阅读量")
    
    # 属于历史表特有的两个字段
    history_id: int = Field(..., alias="historyId", description="历史记录主键ID")
    view_time: datetime = Field(..., alias="viewTime", description="浏览时间")

    # 配置：允许通过别名赋值，同时兼容 ORM 解析
    model_config = ConfigDict(
        populate_by_name=True, 
        from_attributes=True
    )

class HistoryListResponse(BaseModel):
    """历史记录列表返回体"""
    list: List[HistoryItemResponse]
    total: int
    has_more: bool = Field(..., alias="hasMore")

    model_config = ConfigDict(
        populate_by_name=True, 
        from_attributes=True
    )
```

### 2. 异步数据访问层核心实现 (`crud/history.py`)

实现新增/更新、查询列表、单条删除、全部清空 4 个核心底层数据库操作：

```python
# crud/history.py
from datetime import datetime
from sqlalchemy import select, delete, func
from sqlalchemy.ext.asyncio import AsyncSession
from models.history import History
from models.news import News

async def add_or_update_history(db: AsyncSession, user_id: int, news_id: int) -> History:
    """添加或更新浏览历史（实现高频置顶的 Upsert 逻辑）"""
    # 1. 查询该用户针对该新闻是否已有浏览记录
    stmt = select(History).where(History.user_id == user_id, History.news_id == news_id)
    result = await db.execute(stmt)
    existing_history = result.scalar_one_or_none()
    
    if existing_history:
        # 2. 如果存在，直接更新浏览时间为当前时间（触发置顶）
        existing_history.view_time = datetime.now()
        await db.commit()
        await db.refresh(existing_history)
        return existing_history
    else:
        # 3. 如果不存在，插入新纪录
        new_history = History(user_id=user_id, news_id=news_id)
        db.add(new_history)
        await db.commit()
        await db.refresh(new_history)
        return new_history


async def get_user_history_list(db: AsyncSession, user_id: int, page: int, size: int):
    """分页拉取用户的浏览历史列表（包含新闻详情）"""
    # 1. 连表查询：History INNER JOIN News，按 view_time 降序（最新浏览在最前）
    # 同时在 Select 中投影出我们需要的字段（包括 History.id 映射为 history_id）
    stmt = (
        select(
            News.id, News.title, News.description, News.image, News.author, News.category_id, News.views,
            History.id.label("history_id"), History.view_time
        )
        .join(History, History.news_id == News.id)
        .where(History.user_id == user_id)
        .order_by(History.view_time.desc())
    )
    
    # 2. 查询总记录数
    count_stmt = select(func.count()).select_from(History).where(History.user_id == user_id)
    total_result = await db.execute(count_stmt)
    total = total_result.scalar() or 0
    
    # 3. 分页切片
    offset_val = (page - 1) * size
    stmt = stmt.offset(offset_val).limit(size)
    
    # 4. 执行查询，注意这里返回的是 Row 记录行而非纯 ORM 对象
    result = await db.execute(stmt)
    items = result.all()  # 拿到结构化的元组列表
    
    has_more = total > (page * size)
    return total, items, has_more


async def delete_single_history(db: AsyncSession, user_id: int, history_id: int) -> bool:
    """删除单条历史记录（严格校验 user_id 越权）"""
    stmt = delete(History).where(History.id == history_id, History.user_id == user_id)
    result = await db.execute(stmt)
    await db.commit()
    return result.rowcount > 0


async def clear_all_history(db: AsyncSession, user_id: int) -> int:
    """清空某位用户的所有历史记录"""
    stmt = delete(History).where(History.user_id == user_id)
    result = await db.execute(stmt)
    await db.commit()
    return result.rowcount
```

```python
    """
    rows_deleted = await history_crud.clear_all_history(db, current_user.id)
    return {"message": f"清空历史成功，共清除 {rows_deleted} 条记录"}
```

### 4. 笔记补充：业务细节与系统安全“避坑”解析

在记录本节笔记时，请务必写下这两条具有实战和安全意义的开发考点：
1.  **防止“越权删除（IDOR）”漏洞**：
    在删除单条历史记录的 `delete_single_history` 方法中，执行 SQL 时千万不能只写 `where(History.id == history_id)`。
    *   **原因**：如果黑客恶意调用 `/api/history/delete/99`，若后端不校验 `user_id`，黑客就可以随意删除其他任意用户的历史足迹。
    *   **解决**：必须严格绑定当前登录态用户的 ID：`where(History.id == history_id, History.user_id == user_id)`。
2.  **`populate_by_name=True` 的关键作用**：
    当前端向我们发送 POST 请求 `/api/history/add`，请求体为驼峰格式 `{"newsId": 123}`。
    因为我们在 Pydantic Model 中指定了 `news_id: int = Field(..., alias="newsId")` 并设置了 `populate_by_name=True`，FastAPI 一方面能完美接收前端发来的驼峰 `newsId`；另一方面，如果后端在代码内部直接用 Python 的蛇形命名 `HistoryAddRequest(news_id=123)` 进行测试，也完全可以被支持。这使得前后端数据交互极其丝滑。
