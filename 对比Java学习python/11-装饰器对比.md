# 装饰器对比（Decorator vs Proxy/Annotation）

## 一句话定性

Python 装饰器相当于 Java 的**代理模式 + 注解处理器**——`@app.get("/hello/{name}")` 就是在"注册路由"，把路径和函数绑定在一起。

## Python 代码

```python
# 装饰器本质：高阶函数（函数接收函数，返回新函数）
def logger(func):
    def wrapper(*args, **kwargs):
        print(f"调用前")
        return func(*args, **kwargs)
    return wrapper

@logger
def hello(name):
    print(f"Hello {name}")

# @ 语法糖等价于: hello = logger(hello)
```

```python
# 带参数的装饰器（三层嵌套）—— 装饰器工厂
def repeat(n):             # 第一层：收装饰器参数
    def decorator(func):   # 第二层：收原函数
        def wrapper(*args, **kwargs):  # 第三层：包装逻辑
            for _ in range(n):
                func(*args, **kwargs)
        return wrapper
    return decorator

@repeat(3)
def say_hello(name):
    print(f"Hello {name}")

# 等价于: say_hello = repeat(3)(say_hello)
# repeat(3) → 返回 decorator
# decorator(say_hello) → 返回 wrapper
```

```python
# FastAPI 的 @app.get("/path") 底层逻辑（简化版）
class App:
    def __init__(self):
        self.routes = {}
    
    def get(self, path: str):
        def decorator(func):
            # 注册路由：把路径和函数关联起来
            self.routes[path] = func
            return func  # 注意：返回原函数，不是 wrapper！
        return decorator

app = App()

@app.get("/hello/{name}")
async def say_hello(name: str):
    return {"message": f"Hello {name}"}

# 等价于:
# temp = app.get("/hello/{name}")  # 返回 decorator
# say_hello = temp(say_hello)       # decorator 注册后返回原函数
```

## Java 对比代码

```java
// Java 对比：代理模式（静态代理）
public interface Greeting {
    void hello(String name);
}

public class GreetingImpl implements Greeting {
    public void hello(String name) {
        System.out.println("Hello " + name);
    }
}

// 相当于 Python 的装饰器
public class LoggerProxy implements Greeting {
    private Greeting target;
    
    public LoggerProxy(Greeting target) {
        this.target = target;
    }
    
    @Override
    public void hello(String name) {
        System.out.println("调用前");
        target.hello(name);
    }
}

// 使用
Greeting hello = new LoggerProxy(new GreetingImpl());
hello.hello("Alice");
```

```java
// Java 对比：注解（仅元数据，不改变行为）
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Get {
    String path();
}

// Spring 的做法——启动时通过反射扫描注册
@Component
public class HelloController {
    
    @Get("/hello/{name}")
    public String sayHello(String name) {
        return "Hello " + name;
    }
}
```

## 差异表格

| 对比项 | Python 装饰器 | Java 对应 |
|--------|--------------|-----------|
| 本质 | 高阶函数（函数接收函数返回函数） | 代理模式 / AOP 环绕通知 |
| 语法 | `@decorator` 语法糖 | 注解 `@Annotation`（仅元数据） |
| 带参数装饰器 | 三层嵌套函数（工厂模式） | 注解属性 + 反射处理器 |
| 执行时机 | 定义时立即执行替换 | 运行时通过代理调用 |
| 修改行为 | 直接替换原函数引用 | 通过代理对象间接实现 |
| FastAPI/Spring 路由 | `@app.get()` 注册路由返回原函数 | `@GetMapping` 启动时扫描注册 |

## 要点说明

1. **`@` 语法只是语法糖**：`@decorator` 等价于 `func = decorator(func)`，没有魔法。只是写法更简洁。

2. **FastAPI 的 `@app.get()` 不是包装函数，而是注册**：FastAPI 内部把路径和函数登记到路由表里，然后返回原函数本身。所以你的 `say_hello` 函数本身没被修改，只是在路由表中多了一条记录。

3. **装饰器在导入时执行**：当 Python 文件被 import 时，装饰器代码立即执行。不像 Java 注解只是元数据，要到运行时通过反射才被读取处理。

4. **带参数的装饰器 = 工厂模式**：`@app.get("/path")` 这种写法，`app.get("/path")` 先被调用返回一个装饰器函数，然后这个装饰器再作用于下面的函数。所以是两层函数调用。
